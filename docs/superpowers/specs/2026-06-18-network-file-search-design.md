# Дизайн: NetSearch — быстрый поиск файлов по сетевому диску (аналог WizFile)

**Дата:** 2026-06-18
**Статус:** утверждён, готов к планированию реализации

## 1. Цель и контекст

Нужна Windows-программа для быстрого поиска файлов, папок и т.п. по сетевому
диску — аналог WizFile, но работающий по сетевым путям.

**Ключевое техническое ограничение.** WizFile быстрый потому, что читает MFT
(Master File Table) напрямую с локального NTFS-тома. По сети (SMB-шара,
маппированный диск) этот трюк недоступен — нет доступа к MFT удалённого тома.
Поэтому архитектура иная: **обход дерева папок → локальный индекс → мгновенный
поиск по индексу.** Первичный обход занимает время, далее поиск мгновенный.

## 2. Зафиксированные требования

- **Целевой диск:** любой/неизвестный сетевой путь по SMB (`\\server\share`, `Z:\`).
  Нужна поддержка нескольких корневых путей одновременно.
- **Масштаб:** заранее неизвестен → решение масштабируется от десятков тысяч до
  нескольких миллионов записей.
- **Технология:** C# / .NET, GUI на WPF, одиночный `.exe`.
- **Хранилище индекса:** локальный SQLite + загрузка в память (вариант A, см. ниже).
- **Обновление индекса:** кнопка «Обновить» + авто по расписанию (таймер).
- **Поиск:** по имени (подстрока + маски `*?`), фильтры (размер/дата/тип), regex.
  Поиск по содержимому — отдельный режим «по запросу» по текущей выборке.

### Выбранный вариант поискового ядра (A)

Индекс хранится в SQLite на диске. При запуске данные подгружаются в память для
мгновенной фильтрации «по мере ввода» (ощущение WizFile/Everything). Для очень
больших объёмов — переключение на запросы к БД с дебаунсом. Преимущества:
мгновенный отклик, индекс переживает перезапуск, нет повторного медленного
обхода сети при каждом старте.

## 3. Архитектура

Один WPF-десктоп `.exe`, разбитый на независимо тестируемые модули:

```
┌─────────────────────────────────────────────────────────┐
│                     WPF UI (MainWindow)                   │
│   строка поиска · фильтры · таблица результатов · статус  │
└───────────────┬──────────────────────────┬───────────────┘
                │                          │
        ┌───────▼────────┐         ┌───────▼─────────┐
        │  SearchEngine   │         │  IndexManager   │
        │ (фильтрация,    │◄────────┤ (обход дерева,  │
        │  маски, regex,  │  данные │  запись в БД,   │
        │  поиск по тексту)│        │  расписание)    │
        └───────┬─────────┘         └───────┬─────────┘
                │                          │
        ┌───────▼──────────────────────────▼───────┐
        │   IndexStore (SQLite + in-memory кэш)      │
        └───────────────────────────────────────────┘
```

### Модули (единственная ответственность каждого)

- **`IndexStore`** — хранилище. SQLite-файл + загрузка в память при старте.
  API: `BulkUpsert`, `Remove`, `LoadAll`, `QueryDb`. Не знает про UI и обход дисков.
- **`Crawler`** — рекурсивно перечисляет файлы/папки заданного пути, отдаёт записи
  потоком. Изолирован, тестируется на локальной временной папке.
- **`IndexManager`** — оркестрирует обход: запускает `Crawler`, пишет в `IndexStore`,
  ведёт расписание (таймер) и кнопку «Обновить», умеет инкрементальный апдейт.
- **`SearchEngine`** — берёт данные из `IndexStore`, применяет запрос
  (подстрока/маска/regex) и фильтры. Поиск по содержимому — отдельный метод по
  текущему списку результатов.
- **`MainWindow` (+ ViewModel)** — UI: поле поиска с дебаунсом, панель фильтров,
  виртуализированная таблица, контекстное меню, индикатор индексирования.

### Поток данных

Настройки путей → `IndexManager` запускает `Crawler` → записи пишутся в
`IndexStore` (SQLite + память) → пользователь печатает → `SearchEngine` фильтрует
in-memory список → результаты в таблицу. Поиск по содержимому запускается
отдельной кнопкой и читает файлы только из текущей выборки.

## 4. Модель данных (SQLite)

Таблица `entries`:

| Поле | Тип | Назначение |
|------|-----|-----------|
| `id` | INTEGER PK | внутренний id |
| `root_id` | INTEGER | ссылка на корневой путь (источник) |
| `name` | TEXT | имя файла/папки (поиск и показ) |
| `name_lower` | TEXT | имя в нижнем регистре (регистронезависимый поиск) |
| `parent_path` | TEXT | путь родительской папки |
| `is_dir` | INTEGER | 0 — файл, 1 — папка |
| `size` | INTEGER | размер в байтах (для папок 0/NULL) |
| `ext` | TEXT | расширение в нижнем регистре (фильтр по типу) |
| `modified` | INTEGER | дата изменения (Unix-время) |

Таблица `roots`: `id`, `path`, `last_indexed`.

Индексы БД по `name_lower`, `ext`, `modified`, `size`. Полный путь вычисляется как
`parent_path` + `name` (не дублируем, экономим место).

## 5. Индексирование

### Crawler

- Рекурсивный обход через `Directory.EnumerateFileSystemEntries` (потоковый).
- Устойчивость к ошибкам: недоступная папка (нет прав/оффлайн) — логируется и
  пропускается, обход продолжается.
- Ограниченный параллелизм (агрессивный параллелизм по сети вредит) —
  настраиваемое число потоков, по умолчанию небольшое.
- Возвращает записи батчами → `IndexStore.BulkUpsert` в транзакции.

### Инкрементальное обновление

При повторном обходе сверяем `modified`+`size` существующих записей: новые —
добавляем, исчезнувшие — удаляем, изменённые — обновляем. Полный пересбор —
отдельной опцией.

## 6. Поиск

### Режимы по имени (приоритет производительности)

1. **Подстрока** (по умолчанию) — `name_lower.Contains(query_lower)` по in-memory
   списку. Мгновенно.
2. **Маска** `*?` — компиляция в простой матчер (или regex из маски).
3. **Regex** — полноценный, по выбору пользователя, с таймаутом матчинга от
   «зависших» паттернов.

### Фильтры (комбинируются с поиском)

Размер (от/до), дата изменения (период), тип/расширение (например `pdf, docx`),
только файлы / только папки.

### Поиск по содержимому (отдельный режим)

Работает по **текущей выборке** результатов (после фильтра по имени). Читает файлы
потоково, ищет подстроку/regex. Параметры: ограничение размера файла, список
текстовых расширений, параллельное чтение с лимитом. Показывает прогресс,
отменяемый.

### Производительность UI

Дебаунс ввода (~150 мс), фильтрация в фоновом потоке, виртуализация таблицы
(рендерятся только видимые строки) — список из миллионов строк не тормозит.

## 7. Интерфейс (MainWindow)

- **Верхняя панель:** поле поиска (`Ctrl+F`), переключатель режима
  (Подстрока / Маска / Regex), кнопка «Поиск по содержимому», кнопка «Обновить
  индекс» с индикатором прогресса.
- **Панель фильтров** (сворачиваемая): размер от/до, период даты, типы (через
  запятую), чекбоксы «файлы/папки».
- **Таблица результатов** (виртуализированная, сортируемая): Имя · Путь · Размер ·
  Дата изменения · Тип. Статус-бар: «Найдено N из M, индекс обновлён в HH:MM».
- **Контекстное меню по строке:** Открыть · Открыть папку (выделить в Проводнике) ·
  Копировать путь · Копировать имя.
- **Двойной клик** — открыть файл.
- **Настройки** (отдельное окно): список корневых путей (добавить/удалить),
  интервал авто-обновления, лимиты (потоки обхода, макс. размер файла для поиска
  по содержимому, список текстовых расширений).

## 8. Обработка ошибок

- Недоступный путь при обходе → пропуск + лог, обход не падает.
- Сетевой путь оффлайн при старте → работа по последнему сохранённому индексу +
  предупреждение «индекс может быть устаревшим».
- Регистронезависимость и Unicode-имена — корректно (храним `name_lower`).
- Длинные пути Windows (>260) — включена поддержка long path.
- «Зависший» regex — таймаут матчинга.
- БД повреждена/версия изменилась — пересоздание индекса с уведомлением.
- Все фоновые операции (обход, поиск по содержимому) — отменяемы
  (`CancellationToken`), UI не блокируется.

## 9. Тестирование (TDD, по слою на модуль)

- **`IndexStore`** — юнит-тесты на временной SQLite-БД: upsert, удаление, запрос,
  загрузка.
- **`Crawler`** — тесты на временной локальной папке (имитация дерева): полный
  обход, пропуск недоступных папок, корректные размеры/даты.
- **`IndexManager`** — инкрементальное обновление: добавление/удаление/изменение
  файлов между обходами.
- **`SearchEngine`** — подстрока/маска/regex, комбинации фильтров, поиск по
  содержимому, граничные случаи (пусто, спецсимволы, Unicode).
- **UI/ViewModel** — логика дебаунса, формирование запроса из состояния фильтров
  (без окна).
- Сетевой обход проверяется на локальной папке (та же логика) + ручная проверка на
  реальной шаре.

## 10. Поставка

Одиночный `.exe` (self-contained .NET publish). SQLite-файл индекса и настройки —
в `%LOCALAPPDATA%\NetSearch\`.
